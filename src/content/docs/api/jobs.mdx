Some operations don't fit inside a single HTTP request. Exporting 50,000 contacts, importing a CSV with row-level validation, soft-deleting 500 records in one motion, or building a multi-page PDF can all take longer than the API's response budget. For those cases, BuildWorkPro provides an async **jobs** subsystem.

A job is a row in a queue. You create it, poll its status (or wait for the completion webhook in a future release), and download the result artifact when it's done.

## When to use jobs

| Scenario                                            | Synchronous endpoint        | Use a job? |
|-----------------------------------------------------|-----------------------------|------------|
| Pull 25 contacts                                     | `GET /api/v1/contacts`     | No         |
| Export every contact to CSV                          | -                          | Yes -- `csv_export` |
| Bulk-import 5,000 leads from a CSV                   | -                          | Yes -- `csv_import` |
| Soft-delete 200 line items                           | -                          | Yes -- `bulk` (delete) |
| Generate a printable PDF for a bid                   | -                          | Yes -- `pdf` |
| Update a single field on one contact                 | `PATCH /api/v1/contacts/1` | No         |

Anything that may exceed five seconds, touches more than ~100 rows, or needs to produce a downloadable artifact is a job.

## Lifecycle

```text
queued -> running -> succeeded
                  -> partial      (csv_import only)
                  -> failed
                  -> cancelled
```

| Status      | Meaning |
|-------------|---------|
| `queued`    | Created but not yet picked up by a worker. |
| `running`   | A worker has it. Progress is updated periodically. |
| `succeeded` | Finished without errors. Artifact ready (if applicable). |
| `partial`   | `csv_import` only -- some rows succeeded, some failed. Both result and errors artifacts are available. |
| `failed`    | The whole job failed. `errorMessage` carries the reason. |
| `cancelled` | You cancelled it via `POST /api/v1/jobs/{id}/cancel`. Workers also poll status between rows and exit promptly when cancelled. |

`succeeded`, `partial`, `failed`, and `cancelled` are all terminal -- once a job reaches one of these, status doesn't change again.

## API surface

| Method  | Path                              | Scope        | Description |
|---------|-----------------------------------|--------------|-------------|
| `GET`   | `/api/v1/jobs`                    | `jobs:read`  | List jobs (cursor paginated). Filter by `type`, `status`, `resourceType`. |
| `GET`   | `/api/v1/jobs/{id}`               | `jobs:read`  | Status, progress, params, result, error message. |
| `GET`   | `/api/v1/jobs/{id}/download`      | `jobs:read`  | 302 redirect to a 1-hour signed download URL. Available on `succeeded` and `partial`. |
| `GET`   | `/api/v1/jobs/{id}/errors`        | `jobs:read`  | 302 redirect to a 1-hour signed errors-CSV URL (for `csv_import` partials). |
| `POST`  | `/api/v1/jobs/{id}/cancel`        | `jobs:write` | Cancel a `queued` or `running` job. Terminal jobs return 409. |
| `POST`  | `/api/v1/jobs/uploads`            | `jobs:write` | Mint a 15-minute signed PUT URL for uploading a CSV input. |
| `DELETE`| `/api/v1/jobs/{id}`               | `jobs:write` | Soft-delete the job row. Artifacts are auto-purged on TTL expiry. |

Jobs are created indirectly: you call the resource-specific kickoff endpoint (e.g., `POST /api/v1/contacts/exports`, `POST /api/v1/contacts/imports`, `POST /api/v1/bids/{id}/pdf`, `POST /api/v1/contacts/bulk-delete`) and it returns the new job row. From that point on, you interact with the job via the endpoints above.

## Polling cadence

Use the smallest interval that still feels responsive to your workload:

- **Short jobs (< 30s expected):** poll every 2 seconds.
- **Medium jobs (30s - 5 min):** poll every 5 seconds.
- **Long jobs (> 5 min):** start at 5 seconds and back off exponentially up to 60 seconds.

A `job.completed` webhook event ships in a future release; once available, it removes the need to poll at all.

<Aside type="caution">
  Don't poll faster than once per second. The endpoint is rate-limited and aggressive polling can wedge other API calls under the same key.
</Aside>

## Downloading artifacts

Both `/download` and `/errors` return `302` with a `Location` header pointing at a Cloudflare R2 signed URL. The URL is valid for **one hour**; after that you must re-call the redirect endpoint to mint a fresh URL.

Most HTTP clients follow 302 redirects automatically -- you typically don't need to handle the redirect yourself. With curl, use `-L`:

```bash
curl -L \
  -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  -o contacts.csv \
  https://app.buildworkpro.com/api/v1/jobs/job_a1b2c3d4e5f6g7/download
```

The signed URL is host-pinned: BuildWorkPro will only ever 302 you to `*.r2.cloudflarestorage.com`, so there is no open-redirect risk.

Artifacts are reaped after **30 days**. Soft-deleting the job (`DELETE /api/v1/jobs/{id}`) does not delete the artifact immediately -- the TTL sweep handles cleanup.

## CSV import: the errors artifact

`csv_import` is the only job type that can finish in the `partial` state. It means at least one row succeeded **and** at least one row failed. (An all-fail import marks `succeeded` with `created: 0, updated: 0`; consumers must always inspect the `errors` count.)

The errors artifact is a CSV with the schema:

```csv
rowNumber,errorCode,message,sourceJson
2,validation_failed,"email must be a valid address","{""firstName"":""Jane"",""email"":""not-an-email""}"
17,commit_failed,"duplicate primary email","{""firstName"":""John"",""email"":""jane@example.com""}"
```

| Column        | Meaning |
|---------------|---------|
| `rowNumber`   | 1-based, with the header as row 1. The first data row is therefore row 2. |
| `errorCode`   | `validation_failed`, `commit_failed`, or an importer-specific code. |
| `message`     | Human-readable explanation. |
| `sourceJson`  | The original source columns for that row, as JSON. Lets you fix and re-submit just the failed rows. |

Fetch it via `GET /api/v1/jobs/{id}/errors`. The 30-day artifact TTL applies here too.

## Worked example: export contacts

```bash
# 1. Kick off the export. The body filters and column whitelist are optional.
JOB=$(curl -s -X POST https://app.buildworkpro.com/api/v1/contacts/exports \
  -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{ "filters": { "type": "customer" } }' \
  | jq -r '.data.id')

echo "queued ${JOB}"

# 2. Poll until terminal. Backoff is left as an exercise.
while :; do
  STATUS=$(curl -s -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
    https://app.buildworkpro.com/api/v1/jobs/${JOB} | jq -r '.data.status')
  echo "status: ${STATUS}"
  case "$STATUS" in
    succeeded|partial|failed|cancelled) break ;;
  esac
  sleep 2
done

# 3. Download (with -L so curl follows the 302 to R2).
[ "$STATUS" = "succeeded" ] && curl -L \
  -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  -o contacts.csv \
  https://app.buildworkpro.com/api/v1/jobs/${JOB}/download
```

## Worked example: CSV import with errors

```bash
# 1. Get a signed upload URL.
UP=$(curl -s -X POST https://app.buildworkpro.com/api/v1/jobs/uploads \
  -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"contentType":"text/csv"}')

UPLOAD_URL=$(echo "$UP" | jq -r '.data.uploadUrl')
SOURCE_PATH=$(echo "$UP" | jq -r '.data.sourcePath')

# 2. PUT your CSV bytes directly to R2.
curl -X PUT --data-binary @leads.csv \
  -H "Content-Type: text/csv" \
  "$UPLOAD_URL"

# 3. Kick off the import.
JOB=$(curl -s -X POST https://app.buildworkpro.com/api/v1/leads/imports \
  -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  -H "Content-Type: application/json" \
  -d "{
    \"sourcePath\": \"${SOURCE_PATH}\",
    \"mapping\": { \"Name\": \"name\", \"Email\": \"email\", \"Source\": \"source\" },
    \"updateInPlace\": false
  }" | jq -r '.data.id')

# 4. Poll, then -- if partial -- pull the errors CSV.
# (poll loop omitted; see previous example)
curl -L -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  -o leads.errors.csv \
  https://app.buildworkpro.com/api/v1/jobs/${JOB}/errors
```

## Cancellation

Cancel a `queued` or `running` job at any time:

```bash
curl -X POST \
  -H "Authorization: Bearer ${BUILDWORKPRO_API_KEY}" \
  https://app.buildworkpro.com/api/v1/jobs/${JOB}/cancel
```

Workers check status between rows for streaming jobs (`csv_export`, `csv_import`) and exit promptly. For `bulk` and `pdf` -- which are single atomic operations -- cancellation is best-effort: the job exits at the next checkpoint or after the current operation completes.

Cancelling an already-terminal job returns `409 conflict`.
